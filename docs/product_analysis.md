## 一、执行摘要（Executive Summary）

blive-monitor 当前是一个**个人向、零成本、纯静态托管的多平台直播/新作品监控工具**：后端是 GitHub Actions 上每 5 分钟跑一次的 Python 检测脚本（B站官方 API + 抖音页面解析/无头浏览器），状态落地为仓库内 JSON 文件并 push 回 Git，前端是 GitHub Pages 上原生 HTML/JS 的监控看板，直连 GitHub API 读写配置、直连 Actions Secret 写推送配置。

**产品亮点（已做扎实的部分）：**
- 直播检测链路完整且经过多轮打磨（B站官方批量 API、抖音 HTML 多策略兜底、批量失败优雅降级）。
- 通知去重设计成熟（`notify_dedup` 双防线 + 冷却 + 永久去重，解决了「重复推送」这个同类项目最大痛点）。
- 状态持久化与冲突合并（`merge_state.py` 语义合并）在「JSON + Git」这种脆弱载体上做到了相当可靠。
- **日志模块刚刚完成功能性重写**：统一 `type/level/detail/account` 模型、前端搜索/多维筛选/分页/排序/展开/7 天统计、四视图合并为单一主视图——这是本轮已交付的高质量能力。

**核心问题（产品不成立或不可信之处）：**
1. **定位漂移 / 文档与代码严重不符**：`docs/blive-monitor-context.md`、`live-monitor-detection-landscape.md` 把「小红书开播检测」描述为项目最关键的技术突破并声称已端到端验证、有 15 个单测；但实际 `check_status.py` 已于 2026-07-10 **移除**小红书分支，前端无小红书入口、测试无 `test_check_xhs.py`。README 标题仅写「B站/抖音」。小红书是**幽灵功能**——既不在代码里，又占据了文档的核心叙事。
2. **监控「不可自检」**：这是一个监控工具，却没有「监控本身的监控」。无 last_run 心跳、无检测时效告警、无运行成功率/延迟指标。一旦 CI 被跳过/失败、或抖音接口长期风控，前端仍显示旧「updated」时间，用户无从知晓监控已「假死」。
3. **通知可靠性弱**：仅支持「单通道」（5 选 1，不是 5 个并行）；推送失败**无重试、无对账、无重发队列**——失败即丢失该事件；无按房间开关、无关键字过滤、无免打扰时段、无通用 Webhook。
4. **安全硬伤**：前端内置一个**全量 `repo/workflow` 权限的默认 PAT** 并硬编码在公开源码中（虽有注释提示撤销，但仍在生产代码里）。
5. **列表管理能力薄弱**：直播/新作列表只能单条增删、无搜索、无批量、无改名、无导入导出；随名单增长（当前已 20 个账号）体验会迅速劣化。

**结论**：项目底子（检测 + 去重 + 持久化）是好的，但作为「产品」还停留在「极客自用脚本 + 看板」阶段。重构的重心不是重写检测逻辑，而是**补足「可信、可控、可管理、可观测」四块产品能力**，并对平台定位做一刀两断的决策。

## 三、缺口分析（Gap Analysis）

### 3.1 平台覆盖：小红书是「幽灵功能」，定位必须一刀两断
- `check_status.py` L64-66 注释明确：「小红书直播开播检测已于 2026-07-10 移除（无头 Chromium 渲染直播间方案；平台每次开播短链会变、数据中心 IP 触发风控，维护成本高且误判/漏推难根除）」。
- 但 `docs/blive-monitor-context.md` 把小红书描述为「项目最关键的技术突破」「补了行业空白」，并声称有 `check_status.py` 的小红书解析器（约 40KB）、`tests/test_check_xhs.py`（15 passed）、`rooms.json` 含 xhs 条目。实际这些**全部不存在于当前代码**。
- 仓库名 `blive-monitor` + 文档叙事容易让人误以为支持小红书。**这是产品信任的最大隐患**：对外宣称的能力与实际不符。
- **决策点**：要么正式放弃小红书并在全文档/命名对齐（推荐，成本最低），要么投入重做（成本高、且移除理由成立，ROI 存疑）。详见 §七。

### 3.2 通知能力：可用但不可靠、不可控
- **不可靠**：`dispatch_push` 仅返回成功/失败布尔值；调用方在失败分支标记 `pushed_fail` 后**不再重试**。对「开播」这类一次性事件，推送失败 = 用户永久错过。缺乏失败对账与重发队列。
- **不可控（单通道）**：`BLIVE_CONFIG.push.type` 是单一值，同一时刻只能用一个渠道。Server酱 5 条/天额度极易耗尽且无自动降级到其他渠道。
- **不可精细**：全量账号统一推送，无法「只监控 A 不推送 B」「标题含『抽奖』才推」「23:00-08:00 免打扰」。无通用 Webhook，难以接入 Discord/Slack/自有系统。

### 3.3 监控健壮性：缺「监控本身的监控」
- 无 `last_run` 心跳、无「检测时效」指标。前端状态 pill 基于 `status.json.updated`，但这是「上一次成功写入」时间，而非「上一次成功运行」。若 CI 被 GitHub 跳过/失败，前端静默显示陈旧数据。
- 单账号持续 error 无升级告警、无「该房间长期失联」提示。
- 抖音检测走单浏览器上下文，缺多账号限流/轮换策略，数据中心 IP 风控风险随名单扩大而上升（当前 `ENABLE_POST_CHECK` 下每个账号一次 playwright 打开）。

### 3.4 前端体验：直播/新作列表是短板，缺健康仪表盘
- 日志模块已很强；但**直播列表与新作列表管理落后**：仅单条增删、无搜索、无批量、无改名、无导入导出。名单到 20 个已显笨重。
- 无**一屏聚合的健康仪表盘**：在播数、新作数、监控健康度、近期运行指标应同屏呈现，而非散落在各 tab。
- 移动端仅有基础响应式；无 PWA 可安装、无下拉刷新、无骨架屏。暗色已有，i18n 无。

### 3.5 数据与可观测性：有日志无指标
- `history.json` 上限 500 条，仅支撑近 7 天粗粒度统计；无检测成功率、无各账号在线率/开播时长分布、无检测延迟。
- 无数据导出（CSV/JSON），用户无法把监控数据挪作他用（如复盘主播活跃度）。
- 无「监控自身的监控」面板（见 3.3）。

### 3.6 账号 / 权限：内置全权限 Token 是安全硬伤
- `monitor.html` 硬编码一个**全量 `repo/workflow` 权限的默认 PAT**（`ghp_...`），虽注释劝撤销，但**仍在生产源码中**，任何查看页面源码者可得。应改为「无默认 Token、必须由用户自建细粒度 Token」，或在部署时彻底移除。
- 无多用户、无只读分享视图。当前公开 Pages 本身可读 `status.json/rooms.json`（本就公开），编辑才需 Token；但缺乏「生成一个只读看板链接分享给他人」的轻产品化能力。

### 3.7 其它产品化能力（普遍缺失）
- 定时摘要（每日「谁开播了/发了啥」汇总推送）。
- @提及 / 标题关键字提及定向通知。
- 去重策略可视化（用户看不到哪些已被去重、冷却如何）。
- 配置版本/变更历史 UI（rooms.json 变更虽在 Git 历史里，但无前端可追溯）。
- 分组与按组分通知策略。

| **P2-3** | 主播维度画像（开播时段热力图、最佳开播时间建议） | 运营向增值洞察 | M | 数据 |
| **P2-4** | 多渠道故障自动降级（A 失败自动走 B） | 通知永不静默 | M | 通知 |
| **P2-5** | 配置分组与按组分通知策略 | 大名单组织能力 | M | 前端/通知 |
| **P2-6** | 检测策略灰度/可开关（每房间选策略） | 高级用户调优 | S | 直播监控 |

## 五、产品愿景（Product Vision）

> **理想的 blive-monitor**：一个「装好就忘、出事必知、推送可控」的多平台开播/新作监控中枢。
> 它应当：① **可信**——监控自身健康可见、通知失败必重试、状态陈旧必告警；② **可控**——通知能按账号开关、按关键字过滤、按时段免打扰、多渠道并行与降级；③ **可管**——名单批量增删导入导出、变更可追溯、分享只读看板；④ **可观测**——一屏健康仪表盘 + 运行指标 + 数据导出；⑤ **轻盈**——延续零成本、纯静态托管的极客精神，仅在真正需要时引入后端。它不追求做成臃肿的 SaaS，而是把「个人/小团队监控主播动态」这件事做到极致可靠与顺手。

## 六、分阶段重构建议（重构路径）

> 原则：**先止血与可信，再可控与可管，最后扩展与架构**。当前「JSON + Git + CI + 静态前端」架构在阶段 0–3 内**完全撑得住**，无需一开始就上后端。

### 阶段 0 — 定位与安全止血（约 1 周）｜ P0-5、P0-6
- 决策小红书去留并全文档/命名对齐（建议正式放弃，文档改为「B站/抖音」）。
- 移除内置全权限默认 PAT；改为无默认 Token + 强制细粒度提示。
- 清理幽灵文件（如仓库根的 `null` 占位文件等）。
- **产出**：文档与代码一致、无公开全权限凭证。当前架构可撑。

### 阶段 1 — 监控可信化（约 2–3 周）｜ P0-1、P0-2、P1-7
- 引入 `last_run`/运行状态文件，前端健康度 + 时效告警。
- 通知失败重试 + 重发队列 + 结果可见。
- 采集检测成功率/延迟/在线率，提供数据导出。
- **产出**：工具从「可能假死无人知」变为「状态透明、推送必达」。当前架构可撑（新增 JSON 状态文件即可）。

### 阶段 2 — 通知精细化与列表管理（约 2–3 周）｜ P0-3、P0-4、P0-7、P1-4、P1-8
- 按房间开关 + 关键字过滤 + 免打扰；去重可视化；只读分享。
- 直播/新作列表搜索、批量增删、CSV 导入导出、改名。
- 统一健康仪表盘（聚合在播/新作/健康/指标 + 趋势）。
- **产出**：从「极客脚本看板」升级为「可日常使用的管理面板」。当前架构可撑。

### 阶段 3 — 产品化扩展（按需）｜ P1-1、P1-2、P1-3、P1-5、P1-6、P1-9、P1-10、P2-*
- 通用 Webhook、定时摘要、@提及、配置版本 UI、抖音双保险、i18n、PWA、小红书新作（若保留）等。
- **产出**：生态接入与体验拉满。当前架构基本可撑；P1-6 抖音双保险仍在脚本内。

### 阶段 4 — 架构级演进（条件触发，非必须）｜ P2-2
- **何时需要**：出现多用户、实时推送、>100 账号大名单、复杂历史分析、配置版本强一致等需求时。
- **方案**：引入轻量后端（Cloudflare Worker / Fly.io / 云函数）+ 数据库（SQLite/Postgres/Turso），将 JSON 状态文件替换为数据库表，CI cron 替换为常驻或 serverless 定时；前端读写经后端 API（带鉴权）。
- **明确边界**：阶段 0–3 的所有 P0/P1 都可在**现有纯静态 + CI 架构下完成**，不要过早引入后端导致复杂度爆炸。后端是「能力到达天花板时的演进」，不是重构起点。

## 七、待主理人 / 用户确认的关键问题

1. **小红书去留（最高优先）**：正式放弃（推荐，文档对齐为「B站/抖音」）还是投入重做直播+新作监控？（代码已移除且移除理由成立，重做 ROI 存疑。）
2. **后端服务是否引入**：本次重构是否接受「纯静态 + CI」架构延续到阶段 3，仅把后端列为阶段 4 条件触发？还是希望一开始就上后端？
3. **通知渠道优先级**：5 个渠道里用户最常用哪几个？是否要把「多通道并行 + 失败降级」列为 P0（本文暂列 P0-2/P1-4，多通道降级列 P2-4）？
4. **是否需要数据库**：当前 JSON 上限 500 是否够？是否接受「分析深度受 500 条约束」直到阶段 4？
5. **多用户/分享需求**：是否需要「只读分享看板」「多用户」「配置版本管理」？这直接决定阶段 4 是否提前。
6. **推送配置安全边界**：是否接受「前端经 libsodium 写 Actions Secret」的现状，还是希望改为「自建后端代理写 Secret」以彻底避免在浏览器持有 Token？
7. **名单规模预期**：预期监控多少账号？若 >100，P1-6（限流轮换）与阶段 4 需提前。
