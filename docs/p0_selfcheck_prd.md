# P0-1 监控自检心跳 · 产品需求文档（简单 PRD）

> 产品经理：许清楚（Alice）｜ 模块：P0-1「监控自检与心跳」
> 定位：**纯前端、轻量**实现，不新增后端文件、不改 CI workflow
> 关联文档：`docs/product_analysis.md`（P0-1 原始条目：M 复杂度，归属「状态/数据 + 前端」）

---

## 一、产品目标

用户打开页面能**一眼判断**：监控是不是还活着？数据多久没更新了？CI 是不是挂了？——从「陈旧数据静默显示、用户误以为正常」变为「状态透明、陈旧必告警」，重建对工具的基本信任。

---

## 二、用户故事

| # | 角色 | 诉求 | 收益 |
|---|---|---|---|
| US-1 | 普通用户 | 作为用户，我想打开页面就知道上次成功更新是几分钟前，超过阈值时收到醒目告警 | 立刻识别「监控假死」，不再被陈旧数据误导 |
| US-2 | 普通用户 | 作为用户，我想区分「页面/网络加载失败」和「监控数据陈旧」两种异常 | 知道该去查网络还是去查 CI，不误判 |
| US-3 | 维护者 | 作为维护者，我想在顶栏一眼看到监控健康色（绿/黄/红） | 无需进日志就能确认监控是否可信 |

## 三、需求池

### P0（必须有）— 纯前端，数据源 = `status.json.updated`（北京时间）

| 编号 | 需求 | 说明 / 验收标准 |
|---|---|---|
| **P0-1.1** | 顶部健康指示显示「数据新鲜度」 | 复用并增强现有 `#sdot`（顶栏 status-pill），文案为「刚刚更新 / X 分钟前更新 / 已超过 N 分钟·可能停更」，颜色随新鲜度变化 |
| **P0-1.2** | 设定新鲜度阈值（颜色语义） | 建议：**≤10 分钟 → 绿**；**10 < 分钟 ≤ 30 → 黄**；**> 30 分钟 → 红**并明确提示「监控可能已停更，请检查 CI / PAT」。CI 每 5 分钟跑一次，>30 分钟基本可判定 CI 停摆 |
| **P0-1.3** | 显示「上次成功更新时间」+ 相对时间 | 展示 `updated` 绝对时间（北京时间）+ 相对时间（如「12 分钟前」），常驻可见（跨 tab），取代原仅直播 tab 内的「更新于」行 |
| **P0-1.4** | 区分「加载失败」与「数据陈旧」 | ① 加载失败：`status.json` fetch 失败/超时/解析异常 → 红色但文案为「加载失败·无法读取状态」，提示查网络/部署；② 数据陈旧：`updated` 读取成功但新鲜度超阈值 → 红色「可能已停更」。两者文案与排查指向必须不同 |
| **P0-1.5** | 时区正确（复用 `bjNow()` 思路） | 新鲜度 = `Date.now() − parseBeijing(updated)`，其中 `parseBeijing` 用与 `bjNow()` 一致的方式把北京时间字符串转为真实 UTC 毫秒（见 §四），杜绝「差 8 小时」bug |
| **P0-1.6** | 复用而非新增割裂 UI | 整合现有顶栏 `#sdot` + 原「更新于」文案，不另起一套独立状态区；相对时间复用页面既有 `ago()` 思路 |

### P1（应有）— 可选增强，需后端 / Token，**不进入 P0**

| 编号 | 需求 | 说明 / 取舍 |
|---|---|---|
| P1-1 | 查 GitHub Actions 最近运行判断 CI 是否真在跑 | 需用户 Token（`getGhToken()`）+ Actions API。价值：区分「CI 已停」与「CI 在跑但检测异常/被风控」。纯前端 P0 只能判「数据陈旧」，无法判根因。**建议作为后续独立项** |
| P1-2 | 按平台分别的新鲜度（B站 / 抖音） | 需 CI 分别为两平台写时间戳，超出纯前端范围，须改后端写入。可后置 |
| P1-3 | 心跳异常时主动推送通知 | 复用 P0-2 通知可靠性，仅作为触发条件；P0 阶段先只在前端可视告警 |

## 四、实现要点（数据 / 算法，供架构师参考）

新鲜度计算严格走北京时间，与现有 `bjNow()`（monitor.html L493-497）一致：

```js
// 把北京时间字符串 "YYYY-MM-DD HH:MM:SS" 转为真实 UTC 毫秒（复用 bjNow 思路）
function parseBeijing(s){
  var m = String(s).match(/(\d{4})-(\d{2})-(\d{2})[ ](\d{2}):(\d{2}):(\d{2})/);
  if(!m) return null;
  var d = new Date(+m[1], +m[2]-1, +m[3], +m[4], +m[5], +m[6]); // 先按本地解释
  return d.getTime() + (d.getTimezoneOffset() + 480) * 60000;     // +480min ⇒ 真实 UTC 毫秒
}
function freshness(){
  if(!stat || !stat.updated) return {state:'loadfail'};
  var t = parseBeijing(stat.updated);
  if(t == null) return {state:'loadfail'};
  var mins = Math.floor((Date.now() - t) / 60000);
  var state = mins <= 10 ? 'ok' : (mins <= 30 ? 'warn' : 'stale');
  return {state: state, mins: mins, updated: stat.updated};
}
```

- 数据源：`ld()` 已读取 `status.json` 到全局 `stat`（monitor.html L1014、L1024），无需改动读取链路。
- 触发时机：在 `ld()` 成功后调用 `freshness()` 并刷新顶栏 `#sdot` + 健康条（与现有 L1029-1033 的 `sdot` 渲染合并）。

## 五、UI 设计稿

### 5.1 顶栏健康 pill（增强现有 `#sdot`，monitor.html L152 / L1029-1033）

```
┌──────────────────────────────────────────────────────────┐
│ 📡 直播 & 新作监控            ↻  [● 12 分钟前更新]          │
│    --:--:--                      └dot+文案（颜色随新鲜度）   │
└──────────────────────────────────────────────────────────┘
```

- dot class 复用现有 `.dot.on/.err/.ok`，新增 `.dot.warn`（黄）；文案由原来「直播中 N / 有异常 / 已更新」改为新鲜度文案。

### 5.2 常驻健康条（置于 `.overview` 之下，跨 tab 可见；整合原直播 tab 内「更新于」行）

```
┌──────────────────────────────────────────────────────────┐
│ 💚 监控正常 · 上次成功更新 21:20:52（12 分钟前）            │  ← 绿
│ 🟡 数据偏旧 · 上次成功更新 21:02:10（23 分钟前）            │  ← 黄
│ 🔴 监控可能已停更 · 已超过 38 分钟无更新，请检查 CI / PAT   │  ← 红（陈旧）
│ ⚫ 加载失败 · 无法读取状态文件，请检查网络 / GitHub Pages    │  ← 红（加载失败，文案不同）
└──────────────────────────────────────────────────────────┘
```
> 四种状态同一位置互斥展示，避免信息重复。原 `renderLive()` 第 353-354 行「更新于」行可移除，统一由本健康条承担。

### 5.3 颜色语义表

| 状态 | 颜色 | 触发条件 | 顶栏 dot | 用户动作提示 |
|---|---|---|---|---|
| 正常 ok | 🟢 绿 `--green` | 新鲜度 ≤ 10 分钟 | `.ok` | 无需操作 |
| 偏旧 warn | 🟡 黄 `--yellow` | 10 < 分钟 ≤ 30 | `.warn`（新增） | 留意，可能即将停更 |
| 停更 stale | 🔴 红 `--live` | > 30 分钟 | `.err` | 检查 CI / PAT 是否失效 |
| 加载失败 loadfail | ⚫ 灰红 | `status.json` 读取失败 | `.err` | 检查网络 / Pages 部署 |

## 六、待确认问题（需主理人 / 用户拍板）

1. **阈值取值**：>10 分钟转黄、>30 分钟转红是否合适？还是希望更激进（如 >15 分钟即红）？CI 频率 5 分钟，阈值需与之匹配。
2. **是否做 P1-1（Actions 查询）**：纯前端 P0 只能判「数据陈旧」，无法区分「CI 真挂了」还是「CI 在跑但检测异常」。是否要本期就加（需用户 Token）？建议先不做，留作后续。
3. **健康条放置位置**：放在 `.overview` 之下常驻（推荐，跨 tab 可见）/ 仅顶栏 pill / 还是顶栏 + 配置页两份？
4. **「更新于」文案去留**：原直播 tab 内「更新于」行是否移除、统一由常驻健康条承担（避免重复）？
5. **是否要 per-platform 新鲜度**：B站/抖音分开需 CI 分别写时间戳（超出纯前端），本期是否暂缓？
6. **加载失败排查指向**：加载失败时文案应提示「查网络 / GitHub Pages 部署」，还是要引导用户去「配置」页？（注意：读 `status.json` 走 Pages 静态文件，本不需 Token；与 rooms 读取失败要区分。）
